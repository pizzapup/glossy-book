// ### W3's 5 Rules for [Using Aria](https://www.w3.org/TR/using-aria/#NOTES)

// 1. Use the built-in semantics of native HTML elements or attributes [[2.1]](https://www.w3.org/TR/using-aria/#firstrule)
// 2. Try not to change native semantics [[2.2]](https://www.w3.org/TR/using-aria/#secondrule)
// 3. All interactive ARIA controls must be usable with the keyboard [[2.3]](https://www.w3.org/TR/using-aria/#3rdrule)
// 4. Do not use <code>role="presentation"</code> or <code>aria-hidden="true"</code> on a focusable element [[2.4]](https://www.w3.org/TR/using-aria/#4thrule)
// 5. All interactive elements must have an accessible name [[2.5]](https://www.w3.org/TR/using-aria/#fifthrule)

// ### Some Examples of Redundant ARIA [2.13.1](https://w3c.github.io/using-aria/#some-examples-of-redundant-aria)

// Adding default implicit roles to interactive elements listed in the HTML5 Recommendation is a waste of time:
// `<button role="button">press me</button>`

// Adding ARIA state or property attributes in addition to their native HTML counterparts
// is a waste of time:
// `<input type="text" required aria-required="true">`
// `<div hidden aria-hidden="true">`

// Adding ARIA roles and states or properties to long-implemented structural elements is a waste of time:
// `<h1 role="heading" aria-level="1">heading text</h1>`
